# Design Decisions

## Principles

_Distilled from resolved decisions. Stable, load-bearing context._

1. **Framework owns capability, hosts own strategy.** Canto provides durable state, prompt/runtime boundaries, and tool abstractions. Product policy (approval UX, catalog, memory aggression) belongs in Ion.
2. **Recovery at the right layer.** Transient provider retries stay in provider wrappers. Session-backed overflow recovery lives at runtime so compaction can rebuild the next provider request from durable effective history. Provider-level overflow retry is only for request-independent recovery.
3. **Validation before execution.** Preflight phase runs sequentially (hooks, approvals, cross-call validation). I/O phase runs concurrently. No mixing.
4. **Structured state survives compaction.** Compaction preserves actionable state — file tracking, iterative summary updates, structured templates — not just conversation narrative.
5. **Concrete over abstract until proven otherwise.** Don't invent partial interfaces that consumers can't yet use. Keep helpers concrete until a real second consumer forces generalization.
6. **Rewrite over bridge.** In pre-release code, replace a surface wholesale when the design is better. Do not preserve early abstractions with shims unless they directly reduce risk during an active migration.
7. **Typed over anonymous.** Prefer typed payloads (`StepStartedData`, etc.) over `map[string]any` for any public event surface.
8. **Multi-provider consistency.** Error classification and recovery work across all configured providers, not just the first. The wrapper is the source of truth.
9. **Side effects only when meaningful.** Callbacks and hooks fire only when work will actually happen. No side effects on no-op passes.
10. **Separation of context engineering and context governance.** Building prompts is `prompt/`; managing limits, costs, and compaction is `governor/`. Safety categories are `safety/`.
11. **No compatibility debt at 0.0.x.** Simplify and tighten while changes are cheap. A strong base matters more than preserving early abstractions or compatibility scaffolding.
12. **Cross-provider normalization in request shaping.** When providers lack capabilities (thinking, tool-call IDs), adapt in the request pipeline — not in a separate catalog layer.
13. **Stable Prefix Caching:** Compute fingerprints from static prefix + schemas; structural alignment boundaries preserve economics.
14. **Privacy-First Telemetry:** OTel wrappers drop messages by default; opt-in required to prevent PII leaks.
15. **Superstep Graph Durability:** Graph checkpoints at macro-node boundaries, wait states reuse checkpoints, loops aggregate repeated turns.
16. **Metadata-Driven Execution:** Tool scheduling, concurrency, and capabilities are driven by explicit framework metadata, not schema hacks.
17. **Deterministic Replay & Idempotency:** Semantic handle for tool executions and canonical replayer paths to avoid duplicate side-effects.
18. **Harness facade before product adapters.** Canto should provide one clear
    headless runtime/session authoring path; Ion and other hosts should adapt
    product policy to that path rather than rebuilding framework behavior.

Keeping the mechanism-only posture per principle #1.

- [2026-04-10] Runtime scheduling now lives behind a dedicated scheduler substrate. `Runner.ScheduleChild` delegates through `Scheduler`/`ScheduledTask`, and the default scheduler stays local to `runtime` so hosts can swap scheduling policy later without changing child lifecycle semantics. → Scheduled delegation should reuse the same durable child path instead of inventing a second execution stack.

Fixing the substrate now keeps the design simple, faster, and more coherent under scale.
- [2026-04-10] File-reference dedup now records durable internal file-read facts keyed by `ContentRef` identity, and later prompt injections collapse repeats to cached pointers instead of re-embedding the same bytes. → The framework needs one durable read ledger so repeated workspace reads stop inflating context, but those internal records should stay out of child/artifact exports.
- [2026-04-10] `workspace.WorkspaceFS` shipped as the rooted filesystem capability and the read-only bootstrap/tool surfaces now depend on the interface instead of concrete `Root`. → The filesystem substrate needs to be the abstraction boundary now, not a later refactor around the same concrete type.
- [2026-04-10] Session projection snapshots now sit beside replay as durable rebuild checkpoints. `ProjectionSnapshotted` reuses the durable snapshot payload shape, `ProjectionSnapshotter` appends count/time checkpoints, and `Rebuilder` prefers the latest durable snapshot of either kind. → The framework needs a checkpoint layer for cold starts and rebuilds without inventing a separate cache model or diverging from append-only session durability.
- [2026-04-10] Workspace search indexing now sits on top of `WorkspaceFS` and `ContentRef` as a trigram index keyed by rooted workspace paths. `IndexWorkspace` walks the rooted filesystem, `IndexFile` materializes one file at a time, and `SearchIndex` stays interface-first for later algorithm swaps. → Search is part of the same identity-first substrate as file-read dedup, so it should reuse the rooted workspace boundary instead of inventing a second file graph.
- [2026-04-12] Executor-side secret injection now hangs off `x/tools.Executor` via a `safety.SecretInjector`, with request-time secret names kept out of the prompt/tool payload and injected only after env sanitization. → Credentials belong at the execution boundary, not in model-visible context or ad hoc tool helpers; the executor is the right choke point for env assembly, auditing, and sandboxing.
- [2026-04-12] Executor secret injection is complete and tool output compression is now the next executor-side optimization tranche. → Once credentials are injected at the execution boundary, the next leverage point is reducing output volume rather than widening the boundary again.
- [2026-04-12] Tool output compression now shortens only the model-visible combined output and leaves raw stdout/stderr intact. Exact repeated lines collapse to counted summaries and blank runs collapse to one blank line. → Keeps the executor boundary conservative: raw subprocess data stays available, while the prompt-facing payload shrinks enough to matter.
- [2026-04-12] `ESCALATE.md` parsing now lives in `workspace/` as a root-scoped markdown utility that reads triggers, channels, and approval fallback directives from `WorkspaceFS`. → The approval stack should consume workspace-declared escalation policy without entangling notification delivery or auto-mode behavior in the parser itself.
- [2026-04-13] Auto-mode guardrails split across `approval/` and `governor/`: the approval gate owns policy evaluation and circuit-breaker state, while the governor injects breaker status into the prompt context. → The classifier interface should stay with approval policy, but the breaker state needs to be visible to the request builder so the model sees the actual operating mode.
- [2026-04-13] Prompt cache alignment boundaries are now structurally predictable (last system msg, last tool, recent history suffix) and context mutations (like masking) batch operations within that suffix. → Constantly shifting cache markers invalidate the prefix cache turn-over-turn; stable structural boundaries and batch masking preserve the cache economics.

- [2026-04-22] Overflow recovery integrated at `runtime.Runner`, not only provider wrapper. → Provider-level recovery can compact the session but cannot rebuild an already-built request; runtime recovery retries the whole agent turn so context is rebuilt from the compacted session. - [2026-04-23] Cross-provider normalization moved into `llm.TransformRequestForCapabilities`. → Provider/model switching must preserve transcript continuity at the request-shaping boundary, so role rewriting, thinking flattening, tool-call ID normalization, and synthetic missing tool results now live in a reusable `llm` transform helper rather than as ad hoc `prompt.Capabilities()` logic.

- [2026-04-23] Shared turn-state handling extracted into pure agent logic in `agent/turnstate.go`. → The iterator and streaming turn paths were carrying nearly identical usage accumulation, retry/budget handling, and stop-reason logic. Extracting the turn-state reducer keeps side effects at the step boundary while making loop behavior easier to test and evolve ahead of tool-execution refactors.
- [2026-04-24] Phase 5 design policy: design and implement against the known Codex/Claude Code/Pi/Cursor-class coding/service-agent target rather than waiting on Ion consumption friction to dictate each primitive. Canto stays a general-purpose primitives layer — the coding-agent class is a validating target that exercises the full surface, not a scope ceiling. Primitives must be designed generally and rejected if they only make sense for coding agents. Ion validates; it does not gate. `canto-btl6`, `canto-csp2`, and `canto-8cl4` are derivable from the target today and no longer wait on Ion intake; `canto-p73h` then confirms or amends. → The target shape is well-understood from public prior art; blocking runtime and alpha-contract work on consumer friction trades known-good design for a slower feedback loop. Keeping the general-purpose framing prevents the target from narrowing the primitives.
- [2026-04-24] Tool-surface rule for `coding/`: carry only generally-useful primitives that shell + modern CLIs (`rg`, `fd`, etc.) cannot do cleanly. Signal from Claude Code dropping grep/glob tools once shell approval could auto-allow `rg`/`fd` safely, and from Pi being mostly shell + a few high-leverage extras. Things worth carrying: structured multi-edit, workspace-rooted write with identity tracking, executor/output boundary, sandboxing. Things to question: glob, search, list — likely shell-redundant once a shell-command auto-approval classifier exists in the host. `canto-q56s` audits `coding/` against this rule during `canto-3xay`. Most Ion tools are expected to fall in the generally-useful bucket and live in canto; Ion-specific UX/policy tools stay in Ion. → The coding tool surface is easy to over-specify. Shell + rg/fd with a host-side auto-approve classifier covers a lot of ground; tools that duplicate that coverage are dead weight.
- [2026-04-24] Approval-system split between canto and the host. Rule: canto owns it if removing it would force every app to reimplement durability or state-machine plumbing; the host owns it if removing it just means making a product decision. Canto owns: `WaitState` as a session event (pause must live in the append-only log for resume-after-restart), resume contract, `PolicyFunc` composition type, classifier interface, circuit-breaker state machine (state flows into the request builder), `audit/` append-only sink. Host owns: which tools/args/paths require approval, UX/delivery, classifier implementation (including shell heuristics and LLM-judge prompts), tier thresholds, timeout semantics, who decides. Follow-through: `canto-u4vq` is scoped to verifying the classifier seam is sufficient to host an external shell classifier — canto does not ship a default shell classifier even as an overridable one, because its heuristics are opinionated and environment-specific. → Earlier reasoning drifted toward shipping a default classifier in canto "because any coding-agent needs it." That crosses the mechanism/policy line. The seam is shared; the heuristics are not.
- [2026-04-26] Two-phase tool execution finalized in `agent`: preflight runs validation, `PreToolUse` hooks, approvals, and replay checks sequentially; execution then runs metadata-`Parallel` tools concurrently in source-order waves while `Serialized`/unknown tools remain barriers. Tool results are emitted to the session in original assistant call order regardless of completion order, and `ToolStarted` is now appended at the execution boundary rather than during preflight. → Hooks and approvals can depend on deterministic sibling preflight state, I/O can still overlap safely, and durable replay no longer records a tool as started before it actually receives an execution slot.
- [2026-04-26] Approval classifier seam check: host-owned deterministic classifiers should use `approval.PolicyFunc`, not `llm.ClassifierPolicy`, when they need structured request access or HITL escalation. A shell classifier can inspect `approval.Request.Resource`/`Args`, return automated allow for local read-only cases, or return `handled=false` to let the durable approval flow pause for the host. → Canto keeps the state machine, policy composition, and circuit breaker; Ion owns POSIX parsing, allowlists, user trust, UX, and timeout semantics.
- [2026-04-26] Naming is a Canto pre-Ion API gate, not polish. → Pre-alpha public names should describe primitives at call sites: `approval.Gate` owns durable approval gating and breaker state, `safety.Config` is a mode/category policy implementation, `hook.Handler` mirrors Go handler convention, `audit.StreamLogger` wraps an `io.Writer`, and `governor.CompactQueue` aligns with the rest of governor's `Compact*` surface.
- [2026-04-26] Coding tools are individual primitives, not presets. → Canto should not expose `WorkspaceTools`/`FileTools`/`EditTools` helper bundles that imply a blessed coding-agent tool set. Hosts assemble tool registries explicitly. Canto also should not keep built-in glob/grep-style tools when shell, host-owned indexes, or MCP can own discovery. `list_dir` remains because it is basic rooted filesystem access for no-shell hosts.
- [2026-04-26] Privileged instructions are request-prefix only; durable session history is transcript context. → `MessageAdded` events, bootstrap notices, compaction summaries, and working-set breadcrumbs must not replay as mid-conversation `system`/`developer` messages. Effective history demotes privileged roles to user-visible context, and provider-boundary validation rejects privileged messages after transcript messages.
- [2026-04-26] Canto's session log separates transcript, model-visible context, and hidden events. → As a framework, Canto should not decide product policy like Pi's coding-agent messages, but it should provide the generic lanes: `MessageAdded` for transcript, `ContextAdded` for non-privileged model context, and typed lifecycle/handoff/audit events that stay hidden from prompts unless a host explicitly summarizes them.
- [2026-04-26] Effective history entries preserve source markers. → `EffectiveMessages` remains provider-ready messages, while `EffectiveEntries` carries event type and context kind so consumers can distinguish transcript from context without parsing prompt text. Rebuilders should use those markers first and keep content heuristics only for older snapshots.
- [2026-04-26] Stable context placement is a framework primitive. → Providers reward exact stable prefixes, and most agents have durable environment/task/summary context that should sit before the transcript without becoming privileged instructions. `ContextEntry.Placement` marks `prefix` versus `history`; `prompt.History` records the cache prefix boundary; request-time dynamic context is inserted after that stable prefix.
- [2026-04-26] Session effective history owns canonical prompt ordering. → `prompt.History` should not be the only place that reorders stable context; `session.EffectiveEntries` and `EffectiveMessages` must return the same prefix-before-transcript ordering for uncompacted and compacted histories. Request metadata should name units explicitly (`CachePrefixMessages`) and history-entry context metadata should be explicit (`ContextPlacement`) while preserving durable JSON compatibility.
- [2026-04-26] Prompt-cache boundaries must survive downstream request rewrites. → Any processor that appends session history, masks history, injects request-specific context, prepends new system instructions, or normalizes provider capability gaps must update or preserve `CachePrefixMessages`. Export/eval trajectories should include model-visible `ContextAdded` input; hidden lifecycle events may serve as projection cutoffs but remain hidden from prompt history.
- [2026-04-26] Cache-boundary preservation is an API invariant, not a convention. → `llm.Request` owns cache-aware message insertion helpers, Canto built-ins should use them instead of hand-splicing `Messages`, and prompt-builder append operations should land before cache/capability finalizers when those finalizers exist. Framework consumers should have to opt into dangerous direct mutation, not discover cache rules by reading processor internals.
- [2026-04-26] Provider/model handoff prepares copies, not the neutral request. → Built-in providers should call `llm.PrepareRequestForCapabilities` at the send boundary and convert that provider-ready copy to wire format. This matches Pi-style context handoff: a durable neutral context can be replayed through Anthropic, OpenAI-compatible, and back without one API's role/thinking/tool normalization contaminating the next send.
- [2026-04-26] Eval trajectories preserve input provenance. → `RunTurn.Input` stays as flat model messages for existing scorers, but `RunTurn.InputEntries` carries context-vs-transcript markers for consumers that need policy, scoring, or replay semantics. Eval environment setup should be `ContextEntry` by default, with transcript seeds explicit.
